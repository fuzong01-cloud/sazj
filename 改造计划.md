一、项目改造总体目标

我现在接手的是一个已有的农业病害识别项目。当前项目存在源码结构混乱、Notebook 代码混入工程、依赖不规范、README 缺失、部分功能只存在于 PPT/策划书而未实现、代码与 Kaggle/原作者 Demo 粘连严重、前后端未分离、数据库与用户系统缺失、日志监控不足等问题。

本轮改造目标是把它从一个演示型 Demo，逐步改造成一个具备基本工程规范、可部署、可维护、可扩展的农业病害识别平台。

技术方向：

前端：Vue + JavaScript
后端：Python，建议 FastAPI
数据库：本地开发和短期演示默认 SQLite；后续长期部署可切换 PostgreSQL
架构：前后端分离 + MVVM 前端结构
部署：默认适配 Windows Server 轻量云服务器，配置为 2核 CPU、2GB 内存、40GB 存储；Ubuntu 方案降为备选
文档：README.md + HISTORY.md 长期维护

当前启动策略补充：

- 本地后端不再要求先安装 PostgreSQL，默认使用 `backend/sazj.sqlite3`。
- 后端启动统一使用根目录 `start.py`，避免手动拼接 `uvicorn` 和环境变量命令。
- 构建依赖统一使用根目录 `build.py`。
- 旧根目录入口 `app.py`、`webapp.py`、`Procfile`、`setup.sh` 和旧 `requirements.txt` 已清理。

一、部署方向修改

原计划：

优先适配 Ubuntu 2核2G
Nginx + Uvicorn/Gunicorn + PostgreSQL + systemd

现在改为：

优先适配 Windows Server 轻量云服务器
2核 CPU / 2G 内存 / 40G 存储
部署周期：一个月

这意味着：

不用 systemd
不用 Gunicorn
不用 Linux Nginx 作为默认方案
优先使用 Windows 可执行的部署方式

推荐部署结构：

前端：Vue 打包为 dist 静态文件
后端：FastAPI + Uvicorn
数据库：PostgreSQL for Windows
反向代理：可选 IIS / Caddy / Nginx Windows版
进程守护：NSSM 或 Windows 任务计划程序
环境变量：.env 文件
